## SCHEMA_DRIFT_FIXED
- [2026-05-22] Leaderboard "Could not load the leaderboard" on `/embassy/goals` — **forensic rewrite** (feedback 258d60ac, 552a562c, 06d27345-8d11-49d6-a3aa-e41a8e7a825c): Prior diagnoses (lost `GRANT EXECUTE`) and prior fixes (`20271132000000`, `20271149000000`) wrote identical function bodies and grants, yet 500s persisted across three feedback cycles. Forensic re-analysis: a 500 from PostgREST means PostgreSQL raised an exception during execution — a missing GRANT would surface as 403, not 500. Root cause is therefore inside the function body: the prior `get_chapter_ambassador_activity` body called `_building_in_ambassador_chapter_scope(b.id, p_chapter_id)` once per row across seven correlated subqueries (edits/moderation/visits + three `GREATEST` MAX scans + photos). For an active chapter with ~20 members and tens of thousands of `building_audit_logs` rows, that's hundreds of thousands of nested SECURITY DEFINER calls, each doing a CROSS JOIN with a conditional locality EXISTS — comfortably exceeding Supabase's 8s `statement_timeout` for `authenticated` and surfacing as 500. Migration `20271151000000_leaderboard_rpc_set_based_rewrite.sql` rewrites the body set-based: materialise the chapter's building set once into a `scoped_buildings` CTE, then drive every aggregate off a plain `INNER JOIN scoped_buildings` instead of the per-row function. Functionally identical output (all members including zero-point); orders of magnitude faster; no longer trips statement_timeout. Also issues `NOTIFY pgrst, 'reload schema'` so PostgREST refreshes its function-signature cache immediately. Frontend leaderboard query in `MyGoals.tsx` uses `retry: 0` to avoid hammering the DB on transient failures. **`20271151000000` applied (verified live 2026-07-06).**
