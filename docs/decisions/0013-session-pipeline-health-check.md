# 0013 — Session-start pipeline-health check; the nightly AI review does not page

**Status:** accepted (2026-07-22) — fleet-wide policy, template ADR-0026

**Context.** The pipeline merges PRs unattended (ADR-0005) and nobody reads GitHub notifications, so failures rot silently until an agent happens to trip over them. `06-agent-behaviour.mdc` §0 was a *per-task* pre-flight ("Think Before You Build") — it had no notion of session- or pipeline-level health, which is exactly what the charter's principle 9 ("the system repairs and maintains itself on a clock") promised via its `§0` pointer. The gap was not theoretical: the **Nightly** workflow was red on `main` from 2026-07-16 and its `nightly-failure` issue (#1572) accreted comments for days with no one repairing it. Two of the nightly's failure modes were also benign-but-paging: the `review` job is advisory judgment (a Sonnet pass off the PR path, ADR-0006), yet an Anthropic credit exhaustion — or the claude-code-action bot-actor gate rejecting the scheduled `github-actions` trigger — failed the job and tripped the mandatory `alert`, opening a `nightly-failure` issue for a cause that is not a broken `main`.

**Decision.** Two changes, both principle-9 self-repair. (a) `06-agent-behaviour.mdc` §0 gains a **session-start pipeline-health check** run once before the first task: it triages open `nightly-failure` issues / red `main`, stalled PRs, silently disabled scheduled workflows (GitHub disables crons after ~60 days idle), and drifted `main` branch protection — and repairs them before new work begins. (b) The nightly `review` job is made **non-paging**: its `claude-code-action` step gets `continue-on-error: true` (mirroring the on-demand `ai-review.yml` reviewer) so a credit-exhausted or otherwise-errored advisory review no longer fails the job or trips `alert`, and `allowed_bots: '*'` so the scheduled bot trigger passes the actor gate and the review can actually run once credits exist. `show_full_output: true` stays, so the error remains visible in the run log. The `e2e` job is untouched — a red E2E suite must still fail and page, because it is the real gate on `main`.

**Rejected alternative.** A separate scheduled "self-repair" workflow that opens repair issues. It is redundant with the daily agent session that already runs, cannot exercise judgment on *which* stall to repair (re-arm vs. update-branch vs. fix-check), and adds another cron that itself needs watching — the opposite of the goal.
